# ☕ Better Coffee — 让 AI 帮你冲出更美味的咖啡

> 一个为精品咖啡爱好者打造的冲煮记录与 AI 推荐应用，让你每一天的咖啡都比昨天更好。

## 创意来源

作为精品咖啡爱好者，我每天都会手冲或意式萃取一杯咖啡。但有个困扰：

- 前一天调好的参数，第二天就忘了
- 不同豆子需要不同冲煮参数，但不知道怎么调整
- 买了一堆豆子，不知道哪支该先喝（养豆期不同）
- 找到喜欢的口味后，没法系统记录对应的参数

Better Coffee 就是为了解决这些问题而生。它帮你记录每次冲煮，AI 根据你的口感反馈推荐下一次参数，还能根据养豆状态推荐今日该喝的豆子。

## 功能演示

### 🏠 首页：今日推荐 + 养豆分析

[截图：首页]

打开 App 就能看到：
- AI 推荐今日最适合饮用的豆子（基于烘焙日期和养豆状态）
- 所有豆子的养豆状态分析（养豆中 / 最佳风味期 / 建议尽快饮用）
- 最近 3 条冲煮记录

### 🫘 豆子管理

[截图：豆子列表 + 添加豆子弹窗]

记录每支豆子的：
- 名称、产地、品种、处理法
- 烘焙度（浅/中/深）和烘焙日期
- 重量，自动扣减余量
- 备注

系统按烘焙度自动计算养豆期：
- 浅烘 7-14 天
- 中烘 5-10 天
- 深烘 3-7 天

### ☕ 冲煮记录 + AI 推荐

[截图：冲煮记录页 + AI 推荐结果]

选择豆子和冲煮方式（手冲/意式），填写参数：
- 手冲：粉量、水量、比例、研磨度、水温、时间、器具
- 意式：粉量、液重、比例、研磨度、水温、时间、机器

记录评分、风味标签（16 种可选）、改进方向。

点 **AI 推荐**，AI 根据你的反馈给出下一次冲煮的具体参数建议，包括：
- 推荐参数
- 调整说明
- 综合分析

### 📋 冲煮历史 + 复用参数

[截图：历史列表 + 详情弹窗]

- 查看所有冲煮记录
- 点击查看详情
- **复用参数**：一键把历史参数填入新冲煮表单
- **标记最爱**：找到喜欢的口味，绑定到对应豆子

### ⚙️ AI 配置

[截图：设置页]

支持三家 AI 服务商：
- DeepSeek（推荐，便宜效果好）
- 通义千问
- OpenAI

填入 API Key 即可启用真实 AI 推荐。未配置时进入 Demo 模式，返回模拟数据让评委直接体验。

## 技术架构

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Frontend   │    │   Backend    │    │  Database    │
│  Vue 3 + Vite│←→ │ Express API  │←→  │ MongoDB 7   │
│  Mobile-first│    │ Mongoose ODM │    │  Atlas      │
└──────────────┘    └──────────────┘    └──────────────┘
       ↑                    ↑
       │                    │
   Vercel              Render
   (免费)              (免费)
```

| 层 | 技术 |
|---|---|
| 前端 | Vue 3 + Vite + Vue Router + Axios |
| UI | 移动优先响应式 CSS（无 UI 框架，纯手写） |
| 后端 | Node.js + Express + Mongoose |
| 数据库 | MongoDB 7（Atlas 免费档） |
| AI | DeepSeek / 通义千问 / OpenAI |
| 容器 | Docker + docker-compose |
| 部署 | Vercel + Render + MongoDB Atlas（全免费） |

## TRAE 开发心得

整个项目从需求拆解到代码实现，全程使用 TRAE。

### 印象最深的几点

1. **需求拆解清晰**：TRAE 帮我把模糊的"咖啡记录 App"需求拆解成清晰的迭代计划，每个迭代目标明确
2. **并行 agent 加速**：前端、后端、文档三个 agent 并行工作，效率提升数倍
3. **代码质量稳定**：生成的代码遵循最佳实践，模块化清晰，可维护性好
4. **debug 快速**：出现 require 路径错误时，根据控制台报错精确定位修复

### 解决的最大挑战

最有趣的是 AI 集成部分。三家服务商（DeepSeek/通义千问/OpenAI）虽然都兼容 OpenAI API 格式，但在云函数环境中用 Node.js 原生 https 模块调用需要仔细处理超时和响应解析。最终封装了一个统一的 callProvider 函数，三家共用底层实现，只配置不同的 baseURL 和 defaultModel。
